## 1. 项目概述

| 字段 | 值 |
|---|---|
| 项目名称 | U-Gravitation |
| 类型 | 2D/3D 混合操作解谜平台跳跃（Dimension-Shifting Platform Puzzle） |
| 核心卖点 | 知识锁（Knowledge Lock）+ 维度切换（Dimension Shift）— 玩家通过观察与试错推导隐藏的重力规律，并在 2D 与 3D 视角之间切换发现新路径 |
| 目标平台 | PC（Windows / macOS） |
| 引擎 | Godot 4.6, Forward Plus 渲染, Jolt Physics |
| 视角 | 2D 侧视 / 3D 自由视角（关卡内切换） |
| 目标体验 | 误解 → 试错 → 推导 → 顿悟 → 维度跃迁 |

### 1.1 一句话描述

> 在一个重力规律不断变化的世界中，玩家不被告知规则，而是通过跳跃、坠落、观察，甚至切换到第三维度，亲身推导出隐藏的物理法则，利用它们到达终点。

### 1.2 核心设计理念

游戏的重心不是障碍本身，而是让玩家"用身体感知、理解和利用系统规律"。重力不是机关，而是玩家解谜、规划和操作的核心工具。维度切换不是噱头，而是让玩家发现"同一个世界在不同维度下有不同的真相"。

### 1.3 双核心机制关系

```
重力系统（Knowledge Lock）     维度切换（Dimension Shift）
        │                              │
        │  "规则在变化"                 │  "世界有纵深"
        │                              │
        └──────────┬───────────────────┘
                   │
          玩家需要同时理解：
          1. 重力在如何变化？
          2. 这个空间在 3D 下是什么样的？
          3. 两者如何组合利用？
```

### 3.2 重力方向定义

2D 模式：

| 枚举值 | 向量 | 说明 |
|---|---|---|
| DOWN | (0, 1) | 默认，向下 |
| UP | (0, -1) | 反重力 |
| LEFT | (-1, 0) | 向左 |
| RIGHT | (1, 0) | 向右 |

> 注：Godot 2D 中 Y 轴向下为正。

3D 模式：

| 枚举值 | 向量 | 说明 |
|---|---|---|
| DOWN | (0, -1, 0) | 默认，向下 |
| UP | (0, 1, 0) | 反重力 |
| LEFT | (-1, 0, 0) | 向左（X 负方向） |
| RIGHT | (1, 0, 0) | 向右（X 正方向） |
| FORWARD | (0, 0, -1) | 向前（Z 负方向，仅 3D） |
| BACKWARD | (0, 0, 1) | 向后（Z 正方向，仅 3D） |

> 注：Godot 3D 中 Y 轴向上为正，Z 轴向屏幕外为正。3D 模式新增前后方向的重力，这是 2D 模式下不存在的维度，也是维度切换的核心解谜价值。

### 3.3 重力强度参数

| 参数 | 默认值 | 范围 | 说明 |
|---|---|---|---|
| `gravity_strength` | 980.0 | 0 ~ 2000 | 像素/秒²（2D）或 单位/秒²（3D），基础重力加速度 |
| `gravity_direction` | DOWN | 枚举 | 当前重力方向 |
| `transition_duration` | 0.3s | 0 ~ 2.0 | 重力切换过渡时间 |

## 4. 角色控制器规格

### 4.1 移动参数

| 参数 | 值（2D） | 值（3D） | 说明 |
|---|---|---|---|
| `move_speed` | 300.0 px/s | 6.0 m/s | 水平移动速度 |
| `jump_velocity` | -400.0 px/s | 8.0 m/s | 跳跃初速度（沿重力反方向） |
| `max_fall_speed` | 800.0 px/s | 16.0 m/s | 最大坠落速度（终端速度） |
| `coyote_time` | 0.1s | 0.1s | 离开平台后仍可跳跃的宽容时间 |
| `jump_buffer` | 0.1s | 0.1s | 落地前提前按跳跃的缓冲时间 |

### 4.2 输入映射

| 动作 | 按键 | 说明 |
|---|---|---|
| 左移 | A / ← | 2D: 水平左移; 3D: X 负方向 |
| 右移 | D / → | 2D: 水平右移; 3D: X 正方向 |
| 前移 | W / ↑ | 仅 3D 模式: Z 负方向 |
| 后移 | S / ↓ | 仅 3D 模式: Z 正方向 |
| 跳跃 | Space | 沿重力反方向 |
| 维度切换 | Tab | 在允许的区域触发 2D↔3D 切换 |
| 重置当前关卡 | R | 回到出生点 |

### 4.3 重力响应行为

- 角色的"地面"判定随重力方向旋转：重力向下时脚朝下，重力向左时脚朝左
- 跳跃方向始终为当前重力的反方向
- 重力切换时，角色保留当前速度分量，新重力方向的加速度立即生效
- 角色 Sprite/Mesh 随重力方向旋转

### 4.4 3D 模式下的移动规则

- 3D 模式使用 CharacterBody3D，移动从 2 轴扩展为 3 轴
- 重力在 3D 下可以有 6 个方向（上下左右前后）
- 相机为固定的 45° 俯视角或侧视角（不可自由旋转，降低操作复杂度）
- 玩家在 3D 模式下可以走到 2D 模式中"不存在"的纵深位置

## 5. 重力系统规格

### 5.1 GravityManager（全局重力控制器）

职责：管理当前关卡的全局重力状态。同时维护 2D 和 3D 两套重力向量。

```
属性：
  - current_direction: GravityDirection
  - current_strength: float
  - schedule: Array[GravityEvent]    # 时间驱动的重力变化序列

方法：
  - get_gravity_vector_2d() -> Vector2   # 返回 2D 重力向量
  - get_gravity_vector_3d() -> Vector3   # 返回 3D 重力向量
  - _process_schedule(delta)             # 按时间表推进重力变化

信号：
  - gravity_changed(old_dir, new_dir, strength)
```

### 5.2 GravityEvent（重力事件）

```
属性：
  - trigger_time: float       # 触发时刻（秒，相对关卡开始）
  - direction: GravityDirection
  - strength: float
  - transition: float         # 过渡时长
  - repeat_interval: float    # 0 = 不重复，>0 = 周期重复
```

### 5.3 GravityZone（区域重力场）

职责：当角色进入特定区域时，叠加或覆盖局部重力。2D 模式使用 Area2D，3D 模式使用 Area3D。

```
属性：
  - zone_direction: GravityDirection
  - zone_strength: float
  - blend_mode: enum { OVERRIDE, ADDITIVE }
  - priority: int             # 多区域重叠时的优先级

行为：
  - OVERRIDE: 完全替换全局重力
  - ADDITIVE: 区域重力向量 + 全局重力向量
```

### 5.4 GravityResolver（重力合成器）

职责：每帧为角色计算最终生效的重力向量。根据当前维度返回 Vector2 或 Vector3。

```
输入：
  - GravityManager.get_gravity_vector()
  - 角色当前所在的所有 GravityZone

输出：
  - final_gravity: Vector2 | Vector3

规则：
  1. 收集角色所在的所有 GravityZone，按 priority 降序排列
  2. 如果最高优先级 Zone 为 OVERRIDE → 直接使用该 Zone 的重力
  3. 如果为 ADDITIVE → 全局重力 + 所有 ADDITIVE Zone 的重力向量之和
  4. 无 Zone 时 → 使用全局重力
```

## 6. 维度切换系统规格

### 6.1 核心概念

维度切换（Dimension Shift）允许玩家在同一关卡内，将世界从 2D 侧视图切换到 3D 立体视图，反之亦然。灵感来源于 Fez 的视角旋转和 Super Paper Mario 的维度翻转，但与本游戏的重力系统深度结合。

核心设计原则：
- 2D 和 3D 是同一个世界的两种"观察方式"
- 2D 模式下看不到的纵深（Z 轴）路径，在 3D 模式下可见可走
- 3D 模式下被遮挡或分离的平台，在 2D 模式下可能"对齐"成可通行的路径
- 维度切换不是随时可用的，需要在特定区域（DimensionGate）或满足特定条件时触发

### 6.2 维度状态

```
enum Dimension {
  MODE_2D,    # 2D 侧视模式，CharacterBody2D 激活
  MODE_3D     # 3D 立体模式，CharacterBody3D 激活
}
```

### 6.3 DimensionManager（维度切换控制器）

```
属性：
  - current_dimension: Dimension
  - is_transitioning: bool
  - transition_duration: float = 0.8   # 切换动画时长（秒）
  - can_switch: bool                    # 当前是否允许切换

方法：
  - request_switch() -> bool
      # 玩家按 Tab 时调用
      # 检查 can_switch，检查角色位置是否安全
      # 返回是否成功发起切换
  - _execute_switch(target: Dimension)
      # 执行实际切换：冻结输入 → 播放过渡动画 → 同步角色位置 → 切换激活的角色体 → 恢复输入
  - _sync_position_2d_to_3d(pos2d: Vector2) -> Vector3
      # 2D 坐标映射到 3D：x → x, y → y, z → 保持上次 3D 位置的 z 值（或默认 0）
  - _sync_position_3d_to_2d(pos3d: Vector3) -> Vector2
      # 3D 坐标映射到 2D：x → x, y → y, z 被丢弃（投影）
  - _validate_landing(target_pos) -> bool
      # 检查目标维度中该位置是否安全（不在墙内、不在空中无平台处）

信号：
  - dimension_changed(old_dim, new_dim)
  - dimension_switch_failed(reason: String)
```

### 6.4 DimensionGate（维度切换触发区）

部分关卡中，维度切换仅在特定区域内允许：

```
属性：
  - allowed_dimensions: Array[Dimension]  # 在此区域内可切换到哪些维度
  - auto_switch: bool = false             # 进入时是否自动切换（而非按键触发）
  - one_way: bool = false                 # 是否单向（只能 2D→3D 或 3D→2D）

行为：
  - 角色进入 Gate 区域 → can_switch = true
  - 角色离开 Gate 区域 → can_switch = false
  - auto_switch = true 时，进入即触发切换
```

### 6.5 坐标映射规则

2D 和 3D 世界共享 X/Y 坐标，Z 轴是"隐藏维度"：

```
2D 世界                          3D 世界
(x, y)                           (x, y, z)
  │                                │
  │  x = 水平位置                   │  x = 水平位置（同 2D）
  │  y = 垂直位置                   │  y = 垂直位置（同 2D）
  │  z = 不存在                     │  z = 纵深位置（2D 中不可见）
  │                                │
  └────────────────────────────────┘

2D → 3D 映射：
  pos_3d = Vector3(pos_2d.x / SCALE, -pos_2d.y / SCALE, last_z)
  # SCALE = 像素到米的换算比（如 50px = 1m）
  # Y 轴翻转（2D 向下为正，3D 向上为正）
  # Z 保持上次 3D 模式的值，首次默认 0

3D → 2D 映射：
  pos_2d = Vector2(pos_3d.x * SCALE, -pos_3d.y * SCALE)
  # Z 轴被投影丢弃
  # 这意味着：3D 中不同 Z 位置的物体，在 2D 中会"重叠"
```

### 6.6 维度切换的解谜价值

| 场景 | 2D 视角 | 3D 视角 | 解谜效果 |
|---|---|---|---|
| 断裂平台 | 两个平台之间有不可跨越的间隙 | 平台在 Z 轴上有延伸，可以绕行 | 3D 中发现隐藏路径 |
| 视觉对齐 | 两个平台看起来相连 | 实际上在 Z 轴上相距很远 | 2D 的"欺骗"，需要 3D 验证 |
| Z 轴平台 | 完全不可见 | 存在于 Z≠0 的平台 | 必须切换到 3D 才能发现 |
| 重力+维度 | 重力向下，无路可走 | 切换到 3D 后，Z 轴方向有平台 | 重力方向在 3D 中有新的意义 |
| 投影陷阱 | 看起来安全的落点 | 实际上该位置在 3D 中是空的 | 需要在 3D 中确认再切回 2D |

### 6.7 维度切换限制（防止滥用）

- 切换有 1.5s 冷却时间
- 部分关卡只在 DimensionGate 区域内允许切换
- 切换时如果目标位置不安全（卡在墙内），切换被拒绝并给出视觉反馈
- 切换过程中（0.8s 动画期间）角色不可操作

## 7. 维度切换 Godot 实现方案

### 7.1 技术方案概述

采用"双世界共存"架构：2D 世界和 3D 世界同时存在于同一个场景中，通过控制可见性和物理激活状态来实现切换。这比场景切换方案更流畅，且能实现平滑的过渡动画。

```
为什么不用场景切换（change_scene）？
  - 场景切换有加载开销，无法实现平滑过渡动画
  - 两个世界的状态（重力、计时器、区域）需要同步，共存更简单

为什么不用 SubViewport？
  - SubViewport 适合画中画或渲染到纹理，但不适合主游戏世界的切换
  - 物理碰撞在 SubViewport 中隔离，跨 Viewport 同步复杂
  - 直接在主场景中切换可见性和物理层更高效
```

### 7.2 场景节点结构

每个支持维度切换的关卡使用以下结构：

### 7.3 切换流程（伪代码）

```gdscript
# DimensionManager.gd

func request_switch() -> bool:
    if is_transitioning or not can_switch:
        emit_signal("dimension_switch_failed", "不允许切换")
        return false

    var target = Dimension.MODE_3D if current_dimension == Dimension.MODE_2D else Dimension.MODE_2D

    # 计算目标位置
    var target_pos
    if target == Dimension.MODE_3D:
        target_pos = _sync_position_2d_to_3d(player_2d.global_position)
    else:
        target_pos = _sync_position_3d_to_2d(player_3d.global_position)

    # 安全检查：目标位置是否可站立
    if not _validate_landing(target, target_pos):
        emit_signal("dimension_switch_failed", "目标位置不安全")
        # 播放失败反馈（屏幕闪红 + 震动）
        return false

    _execute_switch(target, target_pos)
    return true


func _execute_switch(target: Dimension, target_pos) -> void:
    is_transitioning = true

    # 1. 冻结玩家输入
    _freeze_player_input()

    # 2. 保存当前速度
    var current_velocity = _get_current_velocity()

    # 3. 播放过渡动画（0.8s）
    await _play_transition_animation(current_dimension, target)

    # 4. 切换世界可见性和物理
    if target == Dimension.MODE_3D:
        world_2d.visible = false
        world_2d.process_mode = Node.PROCESS_MODE_DISABLED
        world_3d.visible = true
        world_3d.process_mode = Node.PROCESS_MODE_INHERIT
        player_3d.global_position = target_pos
        player_3d.velocity = _convert_velocity_2d_to_3d(current_velocity)
    else:
        world_3d.visible = false
        world_3d.process_mode = Node.PROCESS_MODE_DISABLED
        world_2d.visible = true
        world_2d.process_mode = Node.PROCESS_MODE_INHERIT
        player_2d.global_position = target_pos
        player_2d.velocity = _convert_velocity_3d_to_2d(current_velocity)

    # 5. 更新重力系统的维度上下文
    gravity_manager.set_dimension(target)

    current_dimension = target
    is_transitioning = false

    # 6. 恢复玩家输入
    _unfreeze_player_input()

    emit_signal("dimension_changed", current_dimension, target)
```

### 7.4 过渡动画方案

切换时的视觉过渡是玩家体验的关键。推荐方案：

```
2D → 3D 过渡（0.8s）：
  t=0.0s  开始：2D 画面正常显示
  t=0.0s  Camera2D 开始 zoom out（缩小 20%）
  t=0.3s  全屏施加径向模糊 shader（强度 0→0.5）
  t=0.4s  关键帧：2D 世界隐藏，3D 世界显示
          Camera3D 从正交投影开始（模拟 2D 视角）
  t=0.4s  Camera3D 开始过渡：
          - 正交 → 透视投影（FOV 0° → 45°）
          - 位置从正面平移到 45° 俯视角
  t=0.7s  径向模糊消退（0.5→0）
  t=0.8s  完成：3D 画面正常显示

3D → 2D 过渡（0.8s）：
  上述过程的逆序
```

Godot 实现要点：
- Camera3D 的 `projection` 属性可以在 ORTHOGONAL 和 PERSPECTIVE 之间切换
- 使用 Tween 对 Camera3D 的 `fov`、`position`、`rotation` 进行插值
- 径向模糊使用 CanvasLayer 上的 ColorRect + ShaderMaterial
- `process_mode = PROCESS_MODE_DISABLED` 会同时禁用物理和渲染处理

### 7.5 物理层配置

使用 Godot 的碰撞层（Collision Layer）隔离 2D 和 3D 物理：

```
2D 碰撞层分配：
  Layer 1: 2D 地形
  Layer 2: 2D 玩家
  Layer 3: 2D 重力区域
  Layer 4: 2D 危险区域
  Layer 5: 2D 终点
  Layer 6: 2D 维度门

3D 碰撞层分配：
  Layer 1: 3D 地形
  Layer 2: 3D 玩家
  Layer 3: 3D 重力区域
  Layer 4: 3D 危险区域
  Layer 5: 3D 终点
  Layer 6: 3D 维度门
```

> 由于 2D 和 3D 物理引擎在 Godot 中是完全独立的，不需要额外隔离。禁用 `process_mode` 即可停止非活跃世界的物理计算。

### 7.6 关卡设计工作流

为支持维度切换的关卡，推荐以下设计工作流：

```
1. 先设计 3D 关卡布局（这是"真实"的世界）
   - 在 Godot 3D 编辑器中摆放 StaticBody3D + MeshInstance3D
   - 包含 Z 轴上的隐藏路径和平台

2. 从 3D 布局推导 2D 布局
   - 将 3D 世界沿 Z 轴投影到 XY 平面
   - Z 轴上不同位置的平台在 2D 中会重叠或消失
   - 手动调整 2D TileMap 使其与 3D 投影一致

3. 设计维度切换点
   - 在需要玩家切换维度的位置放置 DimensionGate
   - 确保切换后的目标位置安全

4. 验证双向一致性
   - 从 2D 切到 3D 再切回 2D，角色位置应一致
   - 检查所有 DimensionGate 位置在两个维度中都可达
```

### 7.7 性能考量

| 关注点 | 策略 |
|---|---|
| 双世界内存占用 | 3D 世界使用低多边形几何体（CSG 或简单 Mesh），原型阶段不使用高精度模型 |
| 非活跃世界的 GPU 开销 | `visible = false` 跳过渲染；`process_mode = DISABLED` 跳过物理 |
| 切换时的帧率波动 | 两个世界始终在内存中，切换只改变可见性，无加载开销 |
| 3D 光照计算 | 原型阶段使用单个 DirectionalLight3D + 简单环境光，不使用 GI |

#### P-13「纵深」— 维度切换入门

- 验证假设：H5
- 关卡类型：混合（2D↔3D）
- 重力配置：DOWN, 固定不变（两个维度均为向下）
- 初始维度：2D
- 2D 视角布局：

  ```
  │  [S]                     [G]   │
  │  ████      ████          ████  │
  │                                │
  │         ～不可跨越间隙～        │
  │                                │
  │████████████████████████████████│
  │                                │
  │        [DG] 维度门              │
  └────────────────────────────────┘
  DG = DimensionGate
  ```
- 3D 俯视布局（从上方看 XZ 平面）：
  ```
       Z轴（纵深）
       ↑
       │
       │    ████████████████
       │    █ 隐藏走廊(Z=3) █
       │    ████████████████
       │         ↕ 连接
  ─────┼─────────────────────→ X轴
       │  [S]████  间隙  ████[G]
       │     (Z=0 主平面)
       │
  ```
- 玩法描述：
  1. 玩家在 2D 模式下看到出生点和终点之间有不可跨越的间隙
  2. 间隙下方有一个发光的 DimensionGate
  3. 玩家走到 Gate 处，按 Tab 切换到 3D 模式
  4. 在 3D 模式下，玩家发现 Z=3 的位置有一条隐藏走廊，可以绕过间隙
  5. 走过走廊后，切换回 2D 到达终点
- 设计目的：第一次维度切换体验。让玩家理解"2D 中不可见的路径在 3D 中存在"
- 环境提示：
  - DimensionGate 有明显的发光边框和"Tab"图标提示
  - 3D 模式下隐藏走廊有微弱的引导光带
  - 首次进入 3D 时，HUD 短暂显示"WASD 移动"提示
- 预期通关时间：30-45 秒
- 关键设计：
  - Gate 位置明显，不需要寻找
  - 隐藏走廊足够宽，3D 操作容错高
  - 只需要一次 2D→3D→2D 的切换

- 3D 俯视布局：

  - 多个 DimensionGate 分布在关卡中
- 预期通关时间：45-75 秒

- 重力配置：DOWN, 固定不变
- 初始维度：2D

- 预期通关时间：60-90 秒
- 关键设计：需要 3-4 次维度切换才能通关，建立切换的流畅感

- 2D 视角布局：

  ```
  │████████████████████████████████│
  │                          [G]   │

  │  [S]                           │

  │                                │
  │████████████████████████████████│
  │         [DG]                   │
  ```

- 3D 俯视布局：

  2. 切换到 3D，发现 Z=2 处有一个天花板平台
  3. 等待重力切换为 UP，角色"坠"到天花板平台上
  4. 在天花板平台上移动到终点正上方

  6. 切换回 2D 触碰终点

  ```
  │                                │
  │████████████████████████████████│

  ```


- 环境提示：
  - 3D 中 Z 轴走廊内有沿重力方向飘动的粒子
  - 走廊入口有发光标记
- 预期通关时间：60-90 秒
- 关键设计：Z 轴重力是全新概念，走廊设计简单直接，让玩家专注于理解新方向

- 关键设计：
  - 多条可行路径
  - 失败回到出生点，出生点安全
  - 这是"终极顿悟"：玩家需要在脑中同时建模重力变化、区域效果、和 3D 空间结构

## 10. 视觉反馈与感知系统

### 10.1 重力方向粒子

| 属性 | 2D 模式 | 3D 模式 |
|---|---|---|
| 类型 | GPUParticles2D | GPUParticles3D |
| 数量 | 30-50 / 场景 | 50-80 / 场景 |
| 行为 | 粒子运动方向 = 当前重力方向 | 同左，但在 3D 空间中分布 |
| 视觉 | 半透明白色小点，不透明度 0.15-0.25 | 半透明发光小球，不透明度 0.2-0.3 |
| 覆盖范围 | 全屏背景层 | 关卡 3D 空间内均匀分布 |

### 10.2 重力切换过渡效果

- 切换瞬间：短暂的屏幕微震（screen shake, 2-3px, 0.15s）
- 切换瞬间：背景色微调（色相偏移 5-10°, 0.3s 过渡）
- 角色 Sprite/Mesh 旋转动画（0.2s, ease-out）

### 10.3 区域重力场视觉

| 元素 | 2D 模式 | 3D 模式 |
|---|---|---|
| 边界 | 半透明发光边框 | 半透明发光立方体/球体边界 |
| 内部粒子 | 密度高于背景，方向 = 区域重力方向 | 同左 |
| OVERRIDE 区域 | 橙色系 | 橙色系 |
| ADDITIVE 区域 | 蓝色系 | 蓝色系 |

### 10.4 维度切换视觉反馈

| 时机 | 效果 |
|---|---|
| 进入 DimensionGate | Gate 边框亮度提升，HUD 显示 "Tab" 图标 |
| 切换成功 | 0.8s 过渡动画（见 7.4 节） |
| 切换失败（位置不安全） | 屏幕边缘闪红 0.3s + 轻微震动 |
| 切换冷却中 | Tab 图标变灰，显示冷却进度环 |
| 3D 模式下 | 场景有轻微的景深效果（DOF），区分于 2D 的平面感 |

### 10.5 角色反馈

- 着地时：微小的挤压动画（squash & stretch）
- 跳跃时：拉伸动画
- 在空中时：根据速度方向轻微倾斜
- 进入区域重力场时：角色周围出现对应颜色的微光
- 维度切换时：角色短暂"闪烁"（2D sprite 淡出 / 3D mesh 淡入）

### 10.6 Z 轴深度提示（2D 模式下）

为了让玩家在 2D 模式下也能获得一些纵深线索：
- 不在 Z=0 平面的物体在 2D 中显示为半透明（不透明度 0.3-0.5）
- 不同 Z 深度的平台有轻微的色调差异（近处暖色，远处冷色）
- 这些提示是微妙的，不会直接告诉玩家答案，但会引发"这个平台好像不太一样"的直觉

## 12. 验证标准与完成定义

### 12.1 原型完成标准（Definition of Done）

| 标准 | 要求 |
|---|---|
| 角色控制（2D） | 移动、跳跃手感流畅，coyote time 和 jump buffer 正常工作 |
| 角色控制（3D） | WASD + Space 操作流畅，重力响应正确 |
| 重力系统 | 四方向（2D）/ 六方向（3D）重力正确生效，角色地面判定随重力旋转 |
| 时间驱动 | GravityManager 按 schedule 正确切换重力，过渡平滑 |
| 区域重力 | GravityZone 的 OVERRIDE 和 ADDITIVE 模式在两个维度中正确工作 |
| 维度切换 | 2D↔3D 切换流畅，坐标映射正确，过渡动画完整 |
| 维度安全检查 | 切换到不安全位置时正确拒绝并给出反馈 |
| 关卡流程 | 18 关线性流程可完整通关，无卡死 |
| 视觉反馈 | 粒子方向正确跟随重力，区域边界可见，维度切换过渡自然 |
| 稳定性 | 无崩溃，无物理穿模，帧率稳定 60fps（含维度切换时） |

### 12.2 验证方法

1. 开发者自测：每关至少通关 3 次，确认无 bug
2. 观察测试（3-5 名测试者）：
   - 不给任何提示，观察玩家行为
   - 记录每关的：首次通关时间、死亡次数、维度切换次数、是否出现"顿悟"表现
   - 记录玩家口头反馈
3. 核心指标：
   - P-01 ~ P-03：90% 玩家 60 秒内通关
   - P-04 ~ P-06：80% 玩家 120 秒内通关
   - P-07 ~ P-09：70% 玩家 120 秒内通关
   - P-10 ~ P-12：60% 玩家 180 秒内通关
   - P-13 ~ P-15：70% 玩家 120 秒内通关（维度切换有新鲜感加成）
   - P-16 ~ P-18：50% 玩家 240 秒内通关
   - 至少 80% 玩家在流程中表现出至少两次明显的"顿悟"反应（重力顿悟 + 维度顿悟）

### 12.3 已知风险与缓解

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| 重力旋转导致角色控制混乱 | 玩家挫败感过高 | 增加 coyote time，降低初期关卡难度 |
| 时间驱动切换节奏不合适 | 玩家无法推导规律 | 预留 schedule 参数可调，测试时迭代 |
| 区域重力 ADDITIVE 合成不直观 | 玩家无法理解合成效果 | P-10 先用 OVERRIDE 建立概念，P-11 再引入 ADDITIVE |
| 粒子提示不够明显 | 玩家忽略环境线索 | 测试时调整粒子密度和不透明度 |
| 维度切换操作不直观 | 玩家不知道何时/何处可以切换 | DimensionGate 视觉明显，首次出现时有 HUD 提示 |
| 3D 操作手感差 | 玩家在 3D 中迷失方向 | 固定相机角度，3D 关卡空间小且路径清晰 |
| 2D/3D 坐标映射错误 | 切换后角色位置不对 | 严格的安全检查 + 大量边界测试 |
| 双世界性能开销 | 帧率下降 | 非活跃世界完全禁用处理，3D 使用低多边形 |
| 维度切换打断游戏节奏 | 切换过于频繁导致烦躁 | 1.5s 冷却 + 关卡设计控制切换次数（每关 2-4 次） |
